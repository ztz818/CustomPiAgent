## 工作流程

### 项目初始化

用户指定项目路径：`workspaces/ass_fang/projects/NN_案件名称/`

目录结构：
```
projects/NN_案件名称/
├── 00_原始材料/      # 证据、合同、文书等
├── 01_证据清单/      # 证据分类编号
├── 02_法律分析/      # 法律关系、争议焦点
├── 03_文书草稿/      # 起诉状、答辩状等
├── 04_演示文稿/      # PPT、图表
└── 00_案件进度.json  # 进度追踪
```

### 第一步：材料整理

遍历 `00_原始材料/`，分类编号：

```markdown
# 证据清单

## 证据1：[证据名称]
- 证据类型：书证/物证/电子证据
- 证明目的：证明XXX事实
- 来源：XXX
- 日期：YYYY-MM-DD
- 关键内容：要点列表
- 真实性：✅无异议 / ⚠️待核实 / ❌存疑
- 关联性：与争议焦点X相关
- 合法性：✅合法取得 / ⚠️待确认

## 证据2：[证据名称]
（同上）
```

保存到：`01_证据清单/证据清单_YYMMDD.md`

### 第二步：法律分析

分析法律关系和争议焦点：

```markdown
# 法律分析：[案件名称]

## 基本案情
- 原告：XXX
- 被告：XXX
- 案由：XXX
- 诉讼请求：列表

## 法律关系
- 主体：原告、被告、第三人
- 客体：标的物/权利
- 内容：权利义务关系

## 争议焦点
1. 焦点一：描述
   - 原告主张：XXX
   - 被告抗辩：XXX
   - 法律依据：条文
   - 证据支持：证据X、证据Y

2. 焦点二：描述
   （同上）

## 时间线
| 日期 | 事件 | 证据 | 法律意义 |
|------|------|------|----------|
| YYYY-MM-DD | 签订合同 | 证据1 | 合同成立 |

## 法律关系图谱
```mermaid
graph LR
    A[原告] -->|合同关系| B[被告]
    B -->|担保关系| C[担保人]
```

## 法律检索
- 适用法律：《XXX法》第X条
- 相关案例：(YYYY)X法民终X号
- 裁判要旨：XXX
```

保存到：`02_法律分析/法律分析_YYMMDD.md`

### 第三步：文书制作

根据分析结果，制作法律文书：

**起诉状模板**：
```markdown
# 民事起诉状

**原告**：XXX
**被告**：XXX

## 诉讼请求
1. 请求判令被告XXX
2. 本案诉讼费用由被告承担

## 事实与理由
（分段叙述，逻辑清晰）

## 证据清单
1. 证据1：证明XXX
2. 证据2：证明XXX

此致
XXX人民法院

原告：XXX
YYYY年MM月DD日
```

保存到：`03_文书草稿/起诉状_YYMMDD.md`

### 第四步：演示文稿

使用 `frontend-slides` 制作演示文稿：

**庭审演示 PPT**：
- 主题：`corporate-clean` / `academic-paper` / `swiss-grid`
- 风格：简洁、专业、逻辑清晰
- 内容：案情概述 → 争议焦点 → 证据展示 → 法律分析 → 诉讼请求

**客户汇报 PPT**：
- 主题：`pitch-deck-vc` / `minimal-white`
- 风格：重点突出、结论先行
- 内容：核心结论 → 风险提示 → 应对方案 → 时间节点

**培训分享 PPT**：
- 主题：`editorial-serif` / `magazine-bold`
- 风格：案例丰富、结构清晰
- 内容：典型案例 → 法律要点 → 实务技巧 → 风险防范

保存到：`04_演示文稿/PPT_主题_YYMMDD.html`

**分发要求**：
- HTML 生成完成后，必须调用内置 HTML 分发服务上传
- 上传 API：`POST /api/html/upload`
- 渲染地址：`/api/html/render/{projectId}`
- 公网链接：`http://frpv2k.itgo.cc:11881/api/html/render/{projectId}`
- 如果 HTML 引用了 CSS/JS/图片，必须一并上传并重写为项目内相对路径
- 不再使用外部 `view-upload` 服务

## 产物分类原则

### 什么是正式产物？
- 法律文书、证据清单、法律分析
- 演示文稿、图表
- 最终交付给客户的文档

### 什么是中间产物？
- 临时脚本（extract_evidence.py、analyze_case.py）
- 调试代码、实验代码
- 中间数据、临时文件
- 错误日志、调试输出

**判断标准**：如果删掉不影响最终交付，就是中间产物，放 `drafts/YYMMDD/`

---

## 硬规则

### 文件输出
- ✅ **正式产物**：项目目录下（`01_证据清单/`、`02_法律分析/` 等）
- ✅ **中间产物**：`drafts/YYMMDD/`（临时脚本、调试文件、实验数据）
- ✅ **日志**：`logs/`
- ✅ **缓存**：`./uv_cache/`、`./npm_cache/`
- ❌ 禁止写入 /root, /tmp, ~/.cache

**中间产物示例**：
```
drafts/260529/
├── 260529_01_extract_evidence.py   # 临时脚本
├── 260529_02_实验分析.py            # 调试代码
├── 260529_03_数据提取.json          # 中间数据
└── 260529_04_错误日志.txt           # 调试日志
```

**正式产物示例**：
```
projects/01_某某案件/
├── 01_证据清单/                    # 正式输出
├── 02_法律分析/                    # 正式输出
├── 03_文书草稿/                    # 正式输出
└── 04_演示文稿/                    # 正式输出
```

### 依赖管理
- ✅ 只用 uv：`UV_CACHE_DIR=./uv_cache uv ...`
- ✅ npm 缓存：`npm_config_cache=./npm_cache npm ...`
- ❌ 禁止用 pip
- ❌ 禁止写入系统缓存目录

### 法律检索
- ✅ 多源验证：至少2个独立信源
- ✅ 引用规范：法律条文、案例编号、裁判要旨
- ✅ 保存证据：关键裁判文书保存到 evidence/
- ❌ 不编造法律条文
- ❌ 不编造案例

### 文书制作原则
- 事实清楚，证据充分
- 逻辑严密，条理清晰
- 格式规范，用语准确
- 繁简得当，重点突出

## PPT 制作规范

### 主题选择

**庭审演示**：
- `corporate-clean` - 企业级简洁风
- `academic-paper` - 学术论文风
- `swiss-grid` - 瑞士网格风
- 特点：简洁、专业、逻辑清晰

**客户汇报**：
- `pitch-deck-vc` - 投资人 pitch 风
- `minimal-white` - 极简白底
- `sharp-mono` - 锐利黑白
- 特点：重点突出、结论先行

**培训分享**：
- `editorial-serif` - 编辑衬线风
- `magazine-bold` - 杂志粗体风
- `bauhaus` - 包豪斯风
- 特点：案例丰富、结构清晰

### 演讲者模式

按 `S` 键启动演讲者模式：
- 当前页预览
- 下一页预览
- 逐字稿（150-300 字/页，口语化）
- 计时器

### 页面布局

**常用布局**：
- Cover - 封面
- Statement - 核心观点
- Two Column - 双栏对比
- Image Hero - 大图展示
- Timeline - 时间线
- Closing - 结束页

### 动效使用

**适度使用**：
- 庭审演示：少用或不用动效
- 客户汇报：适度使用过渡动效
- 培训分享：可用丰富动效
