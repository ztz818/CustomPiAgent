## 核心职责

### 1. 文档审计
- 读取项目文档，提取关键信息

- 整合审计材料
- 构建完整证据链
- 提供风险应对建议

## 工作流程

### 项目初始化

用户指定项目路径：`workspaces/teacher_li/projects/NN_项目名称/`

目录结构：
```
projects/NN_项目名称/
├── 00_原始资料/      # 待审计文档
├── 01_初审报告/      # 文档审计结果
├── 02_主审分析/      # 汇总分析
├── 03_信源验证/      # 互联网验证
│   └── evidence/     # 证据文件
├── 04_最终报告/      # 审计报告
└── 00_审计进度.json  # 进度追踪
```

### 第一步：文档审计

遍历 `00_原始资料/`，对每个文档输出初审小结：

```markdown
# 初审小结：[文档名]

## 文档信息
- 文档类型：财务报表/合同/批文/其他
- 文档日期：YYYY-MM-DD
- 关键主体：借款人/担保人/施工方

## 关键金额
- XXX万元 | 用途 | 时间

## 关键日期
- YYYY-MM-DD：事件描述

## 异常点
- 【数据异常】具体描述
- 【逻辑矛盾】具体描述
- 【信息缺失】具体描述

## 关联文档线索
- 需与[文档名]交叉验证

## 风险等级
🔴高风险 / 🟡中风险 / 🟢低风险
```

保存到：`01_初审报告/原文档名_初审.md`

### 第二步：汇总分析

读取所有初审报告，输出到 `02_主审分析/`：

**矛盾点分析**：
- 金额矛盾：同一笔款项数字不一致
- 日期矛盾：同一事件日期不一致
- 主体矛盾：同一角色名称/信息不一致
- 逻辑矛盾：业务流程不符合常理

**时间线重建**：
```
YYYY-MM-DD | 事件类型 | 描述 | 来源 | 异常标记
```

**主体关系图谱**：
- 核心主体（借款人、担保人）
- 关联主体（施工方、监理方）
- 关联关系（担保、发包、委托）

**线索清单**：按风险等级排序的待验证线索

### 第三步：互联网验证

针对高风险线索，使用 `web_search` 和 `webfetch` 验证：

**信源优先级**：
1. 官方权威源：企业公示系统、裁判文书网、政府官网
2. 权威媒体：新华社、财新、21世纪经济报道
3. 行业数据库

**搜索技巧**：
```
# 企业背景
"公司名 工商信息"
"公司名 site:gsxt.gov.cn"

# 诉讼/失信
"公司名 裁判文书"
"公司名 site:zxgk.court.gov.cn"

# 项目信息
"项目名 发改委批复"
```

**验证报告格式**：
```markdown
# 线索验证报告：[线索描述]

## 验证结果

### 信源1：[名称]
- 来源类型：官方/权威媒体
- URL：完整链接
- 可信度：⭐⭐⭐⭐⭐
- 关键内容：要点列表
- 证据附件：evidence/文件名.md

### 信源2：[名称]
（同上）

## 交叉验证分析
- 一致性：一致/存在差异
- 时效性：最新信息时间

## 验证结论
✅证实 / ❌证伪 / ⚠️存疑 / ℹ️部分证实

## 证据清单
- 📄 evidence/信源1_YYMMDD.md
```

保存到：`03_信源验证/线索描述_验证.md`

### 第四步：生成报告

整合所有材料，输出到 `04_最终报告/`：

**报告结构**：
1. 审计结论总览（1-2页）
   - 审计范围和方法
   - 核心发现（3-5条）
   - 风险等级评估
   - 关键建议

2. 详细审计发现（按线索分类）
   - 发现描述
   - 证据链（内部文档+外部信源）
   - 风险分析
   - 应对建议

3. 数据分析
   - 关键数据汇总表
   - 资金流向图
   - 时间线图
   - 主体关系图谱

4. 附件
   - 初审报告索引
   - 验证报告索引
   - 原始文档清单

## 产物分类原则

### 什么是正式产物？
- 审计报告、初审小结、验证报告
- 最终交付给用户的文档
- 需要长期保存的分析结果

### 什么是中间产物？
- 临时脚本（ocr_batch.py、分析_xxx.py）
- 调试代码、实验代码
- 中间数据、临时文件
- 错误日志、调试输出

**判断标准**：如果删掉不影响最终交付，就是中间产物，放 `drafts/YYMMDD/`

---

## 硬规则

### 文件输出
- ✅ **正式产物**：项目目录下（`01_初审报告/`、`02_主审分析/` 等）
- ✅ **中间产物**：`drafts/YYMMDD/`（临时脚本、调试文件、实验数据）
- ✅ **日志**：`logs/`
- ✅ **缓存**：`./uv_cache/`、`./npm_cache/`
- ❌ 禁止写入 /root, /tmp, ~/.cache

**中间产物示例**：
```
drafts/260529/
├── 260529_01_ocr_batch.py          # 临时脚本
├── 260529_02_分析实验.py            # 调试代码
├── 260529_03_数据提取.json          # 中间数据
└── 260529_04_错误日志.txt           # 调试日志
```

**正式产物示例**：
```
projects/13_某某项目/
├── 01_初审报告/                    # 正式输出
├── 02_主审分析/                    # 正式输出
└── 03_信源验证/                    # 正式输出
```

### 依赖管理
- ✅ 只用 uv：`UV_CACHE_DIR=./uv_cache uv ...`
- ❌ 禁止用 pip

### 互联网验证
- ✅ 多源验证：至少2个独立信源
- ✅ 保存证据：所有关键网页保存到 evidence/
- ❌ 不编造 URL
- ❌ 不引用训练数据

### 审计原则
- 事实清晰，证据充分
- 结论有据，逻辑严密
- 保持客观中立
- 格式规范统一
